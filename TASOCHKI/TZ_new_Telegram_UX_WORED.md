
# Жёсткое ТЗ на новый Telegram UX для WORED

**Дата:** 28.06.2026  
**Версия:** 1.0  
**Статус:** Product/UX specification  
**Основание:** текущая архитектура WORED, Telegram Bot + Command Deck + Daily Pipeline WORED v2.

## 1. Цель

Полностью переработать Telegram UX в проекте WORED так, чтобы Telegram стал быстрым, понятным и надёжным интерфейсом первого уровня, а не смешанным техническим чатом со случайными сценариями.[cite:2][cite:1]

Новый Telegram UX должен решать 4 задачи:

1. Давать мгновенный доступ к основным сценариям трейдера.
2. Показывать компактную и приоритетную информацию без перегрузки сообщениями.
3. Передавать пользователя в Mini App или WebUI, когда задача требует live-контроля или широкого обзора.
4. Исключать смешение пользовательского, аналитического и инженерного контуров.[cite:2][cite:1][cite:3]

## 2. Контекст проекта

WORED уже имеет Telegram UI, AI routing, WebUI на FastAPI, Redis/PostgreSQL и Daily Pipeline с FSM, revision-командами и execution events, поэтому Telegram не должен тащить на себе весь runtime и все детальные экраны.[cite:2][cite:1]

В Daily Pipeline уже зафиксированы активная торговая сессия, 8-часовой session flow, risk modes (`defensive`, `balanced`, `aggressive`), revision-команды (`continue`, `tighten`, `reduce`, `pause`, `close_all`) и session summary schema, что позволяет сделать Telegram тонким управляющим слоем поверх уже существующей логики.[cite:2]

## 3. Продуктовый принцип

Telegram UX обязан быть построен по правилу **"один запрос -> один главный ответ -> один следующий шаг"**.[cite:2][cite:1]

Запрещено:

- смешивать в одном сценарии AI-chat, futures sim, журнал агента, model picker и session control;
- отвечать на пользовательский запрос несколькими несогласованными сообщениями подряд;
- показывать технические ошибки backend в сыром виде;
- использовать Telegram как замену Mini App для live-runtime управления сессией.[cite:2][cite:1]

## 4. Роль Telegram в общей архитектуре

Telegram — это только **контур быстрого доступа**.[cite:1][cite:2]

Он должен покрывать:

- запуск и просмотр активной сессии;
- быстрый просмотр рынка;
- быстрый анализ по монете;
- быстрый просмотр прогнозов;
- быстрый обзор портфеля;
- управление алертами;
- переход в Mini App или WebUI.[cite:2][cite:1]

Он не должен напрямую быть местом для:

- длинных историй execution events;
- детального просмотра всех сделок и всех метрик;
- инженерной диагностики Docker/Redis/Postgres/API-провайдеров;
- выбора моделей и провайдеров пользователем первого уровня.[cite:1][cite:2]

## 5. Информационная архитектура Telegram

## 5.1. Верхний уровень меню

На первом уровне Telegram должны остаться только 6 пользовательских разделов:

1. **Сессия**
2. **Рынок**
3. **Аналитика**
4. **Прогнозы**
5. **Портфель**
6. **Алерты**[cite:2]

Разделы `Система`, `HER Console`, `Журнал агента`, `Выбор модели`, `Model Lab`, `Provider status` не должны находиться в пользовательском первом уровне, потому что Hermes в проекте уже выделен как отдельный технический orchestrator-контур.[cite:1][cite:3]

## 5.2. Главный CTA

Главной кнопкой в Telegram должна стать **Открыть сессию** / **Mini App**, потому что именно там живёт активная runtime-модель Daily Pipeline: FSM, active snapshot, revision-команды, trades и live events.[cite:2]

Если активной сессии нет, главная кнопка должна называться **Старт сессии**.[cite:2]

## 7. Жёсткие правила UX

## 7.1. Один ответ на один intent

На один запрос пользователь должен получать **одно основное сообщение** и, при необходимости, **один набор кнопок**.[cite:2]

Запрещено разбрасывать один ответ на 2-4 несвязанных сообщения подряд.

## 7.2. Приоритет аварийности

Любой ответ по сессии или позициям обязан выводить данные в порядке важности:

1. авария / критический риск;
2. статус сессии;
3. агрегированный PnL и риск;
4. количество позиций;
5. только затем детали.[cite:2]

## 7.3. Telegram не показывает длинный runtime

Если ответ длиннее 10-12 строк или содержит больше 3 сущностей одного типа, Telegram должен дать digest + кнопку `Открыть Mini App` или `Открыть WebUI`.[cite:2][cite:1]

## 7.4. Никаких сырых ошибок

Любая backend ошибка должна переводиться в user-safe формат. Запрещено показывать в Telegram строки вида:

- `UUID object is not subscriptable`
- stack trace;
- traceback;
- сырые SQL / asyncpg / Redis / JSON ошибки.

Пользователь должен получать только нормализованные ошибки вида:

- `Не удалось получить результат сессии.`
- `Сессия временно недоступна.`
- `Не удалось применить команду revision.`[cite:2][cite:1]

## 7.5. Никакого role confusion

Пользовательский бот не должен случайно переключаться между:

- AI-chat;
- futures sim;
- session runtime;
- journal;
- model picker.

Один intent -> один режим -> один тип ответа.[cite:2][cite:1]

## 8. Форматы обязательных ответов

## 8.1. Формат ответа `статус сессии`

Ответ должен быть строго компактным.

### Шаблон

```text
🎯 Активная сессия
ID: b70e6379
Статус: ARMED
План: v5
Режим риска: balanced
Бюджет: 100 USDT
Открыто позиций: 3
Unrealized PnL: +4.20 USDT (+4.2%)
Realized PnL: -1.10 USDT
Max DD: 2.8% / лимит 6.0%
Последняя команда: tighten
```

### Кнопки

- `📱 Mini App`
- `📦 Позиции`
- `⏸ Pause`
- `🛡 Tighten`
- `🛑 Close all`

### Источник данных

- `trading_sessions`
- `session_plans`
- `session_metrics`
- `session_revisions`
- `executed_trades`[cite:2]

## 8.2. Формат ответа `сколько позиций` / `мои позиции`

По умолчанию Telegram не должен показывать полный список всех позиций, если их больше трёх.[cite:2]

### Шаблон digest

```text
📦 Открытые позиции
Всего: 7
LONG BTCUSDT: 6
SHORT ETHUSDT: 1
Общий unrealized PnL: -1.95 USDT
Общая маржа: 90.00 USDT

Риск: warning
```

### Ниже — максимум 3 позиции по приоритету риска

Приоритет показа:

1. ближайшая к ликвидации;
2. самая убыточная;
3. самая большая по марже.[cite:2]

### Кнопки

- `📱 Открыть Mini App`
- `📄 Все позиции`
- `🛑 Close all`

## 8.3. Формат ответа `баланс и PnL`

## 9. Правила кнопок

У каждого ответа Telegram допускается максимум 5 inline-кнопок.[cite:2]

Типы кнопок:

1. **Primary action** — одно главное действие.
2. **Context actions** — 2-3 действия в рамках сценария.
3. **Escape action** — переход в Mini App/WebUI.[cite:2][cite:1]

Запрещено:

- делать 2 одинаково важных primary action;
- показывать пользователю слишком много технических кнопок;
- смешивать кнопки session control и market analysis в одном ответе без явного повода.

## 10. Intent routing и NLP-требования

## 10.1. Роутинг Telegram UX должен быть intent-first

Роутер обязан сначала определять intent, а уже потом выбирать ответный шаблон или модель.[cite:1][cite:2]

Приоритет intents:

1. `session_control`
2. `session_status`
3. `portfolio`
4. `price`
5. `analysis`
6. `forecast`
7. `alerts`
8. `chat`

## 10.2. Regex-first для критичных команд

Следующие сценарии должны идти через regex-first / deterministic parsing, а не через свободный LLM-chat:

- старт сессии;
- pause / continue / tighten / reduce / close all;
- сколько позиций / мои позиции;
- баланс / pnl;
- результат сессии;
- цена монеты;
- открыть Mini App.[cite:2][cite:1]

## 10.3. Worker-модель только как fallback

Для простых продуктовых команд по Telegram нужно по возможности не тратить большие модели, а использовать deterministic routes и worker-tier только как fallback, что соответствует уже существующей модели tiers и regex-first intents в проекте.[cite:1][cite:2]

## 11. Error UX

## 11.1. Нормализованные ошибки

Только следующие классы ошибок допустимы в пользовательском Telegram UX:

- `session_not_found`
- `session_unavailable`
- `invalid_command`
- `invalid_risk_mode`
- `revision_not_allowed`
- `market_data_stale`
- `temporary_backend_error`
- `auth_required`

## 11.2. Шаблоны ошибок

### Сессия не найдена

```text
ℹ️ Активной сессии нет.
Запусти новую дневную сессию для BTCUSDT.
```

Кнопки:

- `🚀 Старт сессии`
- `📊 Рынок`

### Рынок временно недоступен

```text
⚠️ Данные рынка временно недоступны.
Попробуй обновить через несколько секунд.
```

### Команда revision недоступна

```text
⚠️ Команда сейчас недоступна для текущего состояния сессии.
Проверь статус сессии и повтори действие.
```

## 12. Разграничение контуров

## 12.1. Пользовательский Telegram-контур

В пользовательском боте допускаются только:

- рынок;
- аналитика;
- прогнозы;
- портфель;
- алерты;
- сессия.[cite:2]

## 12.2. Hermes/admin-контур

В отдельном техническом контуре должны жить:

- model routing;
- provider status;
- quota/token accounting;
- Docker/runtime health;
- collector health;
- Redis/Postgres status;
- debug/logs.[cite:1][cite:3]

Эти сущности нельзя выводить в основной Telegram UX первого уровня, потому что они относятся к инженерному, а не пользовательскому потоку управления.[cite:1]

## 13. Связь Telegram с Mini App и WebUI

Telegram должен быть связан с Mini App и WebUI жёстко и предсказуемо.[cite:2][cite:1]

### Правила переходов

- Всё, что связано с активной сессией runtime -> Mini App.
- Всё, что связано с историей, review, long-form analytics, scorecards -> WebUI.
- Всё, что связано с быстрыми командами и digest -> Telegram.[cite:2][cite:1]

### Минимальные deep-links

Нужно поддержать кнопки:

- `Открыть Mini App`;
- `Открыть обзор сессии`;
- `Открыть прогнозы`;
- `Открыть портфель`;
- `Открыть алерты`.[cite:2][cite:1]

## 14. Acceptance criteria

Задача считается принятой, если выполнены все пункты ниже.

### 14.1. Пользовательский UX

- Пользователь за 1-2 тапа попадает в нужный сценарий: Сессия, Рынок, Аналитика, Прогнозы, Портфель, Алерты.[cite:2]
- Telegram не выдаёт длинные полотна текста для live-runtime сценариев.[cite:2]
- Telegram не показывает сырые backend ошибки.[cite:2][cite:1]
- Ответ на `статус сессии`, `мои позиции`, `баланс и PnL`, `результат сессии` стабилен и шаблонизирован.[cite:2]

### 14.2. Архитектурный UX

- Session runtime перенесён в Mini App как основной live-контур.[cite:2]
- WebUI используется для истории, review, прогнозов и настройки.[cite:1][cite:2]
- Hermes/admin-контур отделён от пользовательского Telegram UX.[cite:1][cite:3]

### 14.3. NLP и routing

- Ключевые команды session/portfolio/market идут через deterministic parsing.[cite:2][cite:1]
- Worker-tier используется только как fallback для неочевидных формулировок.[cite:1][cite:2]
- Analyst/Premium не тратятся на простые ответы Telegram первого уровня.[cite:1][cite:2]

## 15. Приоритет внедрения

### Phase 1 — Telegram cleanup

- убрать из первого уровня `Система`, `HER Console`, `Журнал агента`, `Выбор модели`;
- внедрить новое меню из 6 разделов;
- внедрить жёсткие шаблоны ответов для Session/Portfolio/Market.[cite:2][cite:1]

### Phase 2 — Session-first UX

- перевести все live session сценарии на Mini App;
- в Telegram оставить только summary + control buttons.[cite:2]

### Phase 3 — Error-safe routing

- нормализовать ошибки;
- внедрить regex-first на ключевые команды;
- убрать случайные AI-chat ответы в продуктовых сценариях.[cite:2][cite:1]

### Phase 4 — Admin split

- вынести Hermes/admin функции в отдельный контур и отдельный набор команд.[cite:1][cite:3]

## 16. Definition of Done

Новый Telegram UX считается завершённым, если:

- бот стал предсказуемым по сценариям;
- ключевые команды работают без role confusion;
- ответы сжаты, информативны и безопасны;
- Telegram больше не используется как хаотичный универсальный интерфейс;
- Mini App и WebUI встроены как естественные продолжения Telegram сценариев;
- инженерные функции физически и логически отделены от пользовательского опыта.[cite:2][cite:1][cite:3]
